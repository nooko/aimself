# 第六章：避坑提醒

## 不要做的事——详细解释

### 1. 不要从机器学习经典算法开始

**为什么**：

```
传统学习路径（适合学生，不适合你）：
线性回归 → 逻辑回归 → SVM → 决策树 → 随机森林 → XGBoost
→ CNN → RNN → LSTM → Attention → Transformer → GPT

这条路要 12+ 个月，而且前面大部分对 LLM 应用帮助极小。

推荐路径（适合有工程经验的转型者）：
PyTorch 基础 → Transformer → LLM → 应用
需要时回头补特定知识点
```

**例外情况**：
- 做推荐系统 → 需要学 XGBoost/LightGBM
- 做表格数据预测 → 传统 ML 可能比深度学习好
- 做异常检测 → Isolation Forest 等经典算法有用

---

### 2. 不要陷入数学推导

**多深够用**：

| 概念 | 需要的理解深度 | 不需要的 |
|------|--------------|---------|
| 梯度下降 | 知道"沿着最陡方向下山" | 不需要证明收敛性 |
| 反向传播 | 知道"链式求导分配梯度" | 不需要手推每层公式 |
| Attention | 知道"Q×K^T 算相似度，×V 取信息" | 不需要推导信息论解释 |
| LoRA | 知道"低秩分解减少参数" | 不需要证明为什么低秩有效 |
| 量化 | 知道"FP16→INT4 压缩精度换空间" | 不需要推导最优量化方案 |

**判断标准**：
- 能向同事解释清楚这个技术在干什么 ✓
- 能决定什么场景用什么技术 ✓
- 能复现论文中的数学推导 ✗（不需要）

---

### 3. 不要追求看完所有论文

**精读清单（5-10 篇够用）**：

```
必读（理解原理）：
1. Attention Is All You Need (2017) — Transformer 原文
2. LoRA (2021) — 微调方法
3. RAG (2020) — 检索增强生成

推荐读（开阔视野）：
4. LLaMA 系列论文 — 理解模型设计选择
5. vLLM/PagedAttention (2023) — 推理优化
6. ReAct (2022) — Agent 方法论
7. Chain-of-Thought (2022) — 推理能力

选读（按需）：
8. DPO/RLHF — 对齐方法
9. Mixture of Experts — 稀疏架构
10. FlashAttention — Attention 加速
```

**读论文的正确方式**：
```
1. 只读 Abstract + Introduction + 方法部分
2. 跳过证明和消融实验细节
3. 重点看"这个方法解决什么问题"和"核心思路是什么"
4. 配合代码实现一起看
5. 不理解的跳过，做完项目回来再看
```

### 4. 不要只学不练

**每个阶段的最低实践要求**：

```
阶段1：跑通 MNIST（哪怕是复制代码改改）
阶段2：手写 nanoGPT（跟着视频也行）
阶段3：微调一个模型（哪怕只有 50 条数据）
阶段4：搭建一个 RAG Demo（哪怕只有 10 个文档）
阶段5：vLLM 部署一个模型（跑通即可）
阶段6：端到端完整项目

"跑通" > "完美"
"有 Bug 但能用" > "设计完美但没写代码"
```

---

### 5. 不要被框架绑架

**LangChain 的问题**：

```
优点：
- 抽象层完整，Demo 快
- 社区大，教程多

缺点：
- 过度抽象，debug 困难
- 性能开销大
- 版本频繁变化，今天的代码明天可能不跑
- 很多场景用不到它的复杂功能

建议策略：
├── 学习时：了解 LangChain 概念，理解 RAG/Agent 模式
├── 原型验证：可以用 LangChain 快速搭
└── 生产环境：自己写核心逻辑，用 LangChain 的思路但不用它的代码

实际上 RAG 核心代码不超过 100 行：
embed → store → retrieve → prompt → generate
自己写更可控、更可调试。
```

---

### 6. 不要忽视小模型

**小模型的力量**：

```
误区：只有 GPT-4 / Claude 3.5 才够用
现实：7B-14B 模型微调后在垂直场景能打平甚至超越通用大模型

原因：
- 通用大模型知识太广，在特定领域反而不够深
- 微调注入领域知识后，小模型在该领域专家级
- 部署成本：7B 模型一张 4090 就能跑，大模型要数万元/月 API 费用

案例：
- 游戏 NPC 对话：微调 Qwen2.5-7B 效果 > 通用 GPT-4（更懂游戏世界观）
- 代码补全：CodeQwen-7B 微调后 > 通用 Claude（更懂项目代码风格）
- 客服分类：微调 1.5B 模型就够（简单任务不需要大模型）

选型指南：
├── 简单分类/提取：1.5B-3B
├── 对话/问答：7B
├── 复杂推理/生成：14B-32B
└── 通用场景（不想微调）：API 调用大模型
```

## 容易踩的坑——详细分析

### 显卡焦虑

**现实情况**：

| GPU | 显存 | 能做什么 | 价格(参考) |
|-----|------|---------|-----------|
| RTX 4060 | 8GB | 推理 7B INT4，微调 1.5B | ~2500 |
| RTX 4070 Ti | 12GB | 推理 7B FP16，微调 3B | ~4500 |
| RTX 3090/4090 | 24GB | 推理 14B，微调 7B LoRA | ~6000-14000 |
| 2×4090 | 48GB | 推理 32B，微调 14B | ~28000 |
| 云 A100 40GB | 40GB | 几乎所有任务 | ~5-15元/小时 |

**建议**：
```
已有 3060/3070 (8-12GB)：
→ 先用着，INT4 量化跑 7B，微调用 QLoRA
→ 不够再租云 GPU

已有 3090/4090 (24GB)：
→ 足够完成 90% 学习内容
→ 只有训 14B+ 或不量化时才需要更多

没有 GPU：
→ 方案1：Google Colab 免费版（T4 16GB，够入门）
→ 方案2：AutoDL/恒源云 租 A100（~5元/小时）
→ 方案3：先不买，学完前3个阶段再决定
```

---

### 数据焦虑

**真相**：

```
微调不需要百万条数据！

数据量参考：
├── 语气风格微调：100-500 条高质量对话
├── 特定任务微调：500-2000 条
├── 领域知识注入：1000-5000 条
└── 通用能力提升：10000+ 条（这才需要大量数据）

质量 >> 数量：
- 50 条人工精心写的数据 > 5000 条 GPT 随意生成的
- 标注一致性比数量更重要
- 数据多样性比数据量更重要

数据构造技巧：
1. 人工写 30-50 条种子数据（定义质量标准）
2. 用 GPT-4/Claude 扩增（给种子数据做 few-shot）
3. 人工审核筛选（去掉不合格的）
4. 测试集单独留出（不少于 50 条）
5. 迭代：根据模型表现不好的 case 补充数据
```

### 框架焦虑

**不要一开始就上**：

```
不需要的（后面再学）：
├── DeepSpeed — 分布式训练框架（单卡用不到）
├── Megatron-LM — 超大模型训练（做应用不需要）
├── Ray — 分布式计算（小规模用不到）
├── FSDP — 全分片数据并行（单卡用不到）
└── Triton(OpenAI) — 自定义 GPU kernel（太底层）

先学这些就够：
├── transformers — 模型加载和使用
├── peft — LoRA 微调
├── trl — 对齐训练（SFT/DPO）
├── vLLM — 推理部署
└── sentence-transformers — Embedding
```

---

### 过度工程

**正确的迭代顺序**：

```
第一版：能跑就行
├── 硬编码 Prompt
├── 直接调用 API
├── print 调试
└── 本地测试

第二版：基本可用
├── Prompt 模板化
├── 加入 RAG
├── 基础错误处理
└── 简单评估

第三版：生产就绪
├── 服务化部署
├── 监控告警
├── 限流降级
├── 完善评估体系

常见错误：
✗ 还没跑通就想着微服务架构
✗ 数据只有 100 条就纠结分布式训练
✗ Demo 阶段就上 K8s
✗ 还不确定方案可行就优化性能
```
