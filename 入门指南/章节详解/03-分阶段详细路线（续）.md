### Transformer 架构全解

```
输入文本: "The cat sat"
    ↓
[Tokenizer] → Token IDs: [464, 3797, 3332]
    ↓
[Token Embedding] + [Position Encoding] → 每个 token 变成向量
    ↓
[Transformer Block × N 层]
    ├── Multi-Head Self-Attention (核心！)
    ├── LayerNorm
    ├── Feed-Forward Network (MLP)
    └── LayerNorm
    ↓
[Output Head] → 预测下一个 token 的概率分布
    ↓
"on" (概率最高的下一个词)
```

### Self-Attention 详解

**直觉理解**：

每个 token 在"看"所有其他 token，决定该关注谁。

```
"The cat sat on the mat"

当处理 "sat" 时：
- "sat" 强烈关注 "cat"（谁在坐？）
- "sat" 适度关注 "on"（坐在哪？）
- "sat" 较少关注 "The"（不太相关）
```

**数学公式**（很简单）：

```
Attention(Q, K, V) = softmax(Q × K^T / √d) × V

其中：
Q = 查询矩阵 (Query) —— "我在找什么信息？"
K = 键矩阵 (Key)     —— "我能提供什么信息？"
V = 值矩阵 (Value)   —— "我的具体内容是什么"
d = 向量维度          —— 缩放因子防止数值过大

游戏工程师类比：
Q×K^T ≈ 计算相似度（就像你用点积判断两个向量方向是否接近）
softmax ≈ 归一化成概率（让权重加起来=1）
×V    ≈ 加权聚合（根据权重取信息）
```

**Multi-Head**：同时用多组 Q/K/V 从不同角度关注。类似多个"注意力通道"并行工作。

```python
# 简化的 Attention 实现
def attention(Q, K, V, mask=None):
    d_k = Q.shape[-1]
    scores = torch.matmul(Q, K.transpose(-2, -1)) / math.sqrt(d_k)
    if mask is not None:
        scores = scores.masked_fill(mask == 0, float('-inf'))
    weights = torch.softmax(scores, dim=-1)
    return torch.matmul(weights, V)
```

### 位置编码（Position Encoding）

**为什么需要**：Attention 本身不关心顺序（"猫吃鱼"和"鱼吃猫"对 Attention 一样），需要位置信息告诉模型 token 的顺序。

**RoPE（Rotary Position Embedding）**——当前主流方案：

```
核心思想：把位置信息编码为旋转角度
- 位置 m 的 token 向量旋转 m×θ 角度
- 两个 token 的 Attention 分数自然包含相对位置信息
- 支持外推到训练时没见过的长度

你的类比：
就像图形学里的旋转矩阵，用旋转角度编码位置关系。
cos/sin 的周期性让模型能感知距离。
```

### Decoder-Only 与自回归生成

```
Decoder-Only 架构（GPT/LLaMA/Qwen 都是这个）：

训练时：
输入: [The, cat, sat, on]
标签: [cat, sat, on, the]   ← 每个位置预测下一个 token
使用 Causal Mask：每个 token 只能看到前面的 token（不能偷看后面）

生成时：
[The] → 预测 "cat"
[The, cat] → 预测 "sat"
[The, cat, sat] → 预测 "on"
...逐个 token 生成，直到输出 <EOS>
```

### Tokenizer（BPE）

```
BPE (Byte Pair Encoding) 做什么：
把文本切成"子词"单元

"unhappiness" → ["un", "happiness"]  或  ["un", "happ", "iness"]

为什么不按字符/按单词切：
- 按字符：序列太长，信息密度低
- 按单词：词表太大，无法处理新词
- BPE：折中方案，常见词完整保留，罕见词拆成子词

对 token 的感知很重要：
- GPT-4 中 "hello" = 1 token，"你好" = 2-3 tokens
- 一般 1 token ≈ 0.75 个英文单词 ≈ 0.5-1.5 个中文字
- 上下文窗口 128K tokens ≈ 约 10 万字中文
```

### KV-Cache 原理

**这是工程优化的核心，你的强项领域**：

```
问题：自回归生成每个新 token 时，要重新计算所有之前 token 的 K 和 V
解决：缓存已计算的 K/V，新 token 只需计算自己的 Q/K/V

类比：
就像 Redis 缓存查询结果，避免每次都查数据库。
每生成一个新 token，只需要计算这一个 token 的 K/V，
然后把它追加到缓存里。

显存占用计算（LLaMA-7B 为例）：
- 32 层 × 32 头 × 128 维 × 2(K和V) × 2字节(FP16) = 每 token ~1MB
- 4096 tokens 上下文 ≈ 4GB KV-Cache
- 这就是为什么长上下文需要大量显存
```

### 4.2 微调实战

#### 为什么需要微调

```
API 调用 vs 微调的选择：

用 API（不微调）适合：
- 通用任务（翻译、摘要、问答）
- 数据不敏感
- 快速验证想法

微调适合：
- 特定语气/风格（NPC 对话）
- 特定格式输出（游戏协议）
- 数据敏感不能传到外部
- 需要本地部署
- 降低推理成本（小模型微调后效果够用）
```

#### LoRA 原理简述

```
问题：全量微调 7B 模型需要 ~56GB 显存（不现实）
解决：LoRA — 只训练一小部分新增参数

原理：
原始权重 W (frozen，不动)
新增：ΔW = A × B (A: d×r, B: r×d, r << d)
输出：y = (W + ΔW) × x = W×x + A×B×x

r 通常取 8-64，参数量只有原模型的 0.1%-1%
一张 4090 (24GB) 就能微调 7B 模型

类比：
不修改游戏引擎源码(W)，而是写一个 Mod/Plugin(ΔW)。
Mod 很小但能大幅改变行为。
```

#### 微调完整流程

```python
from transformers import AutoModelForCausalLM, AutoTokenizer
from peft import LoraConfig, get_peft_model
from trl import SFTTrainer, SFTConfig

# 1. 加载基座模型
model_name = "Qwen/Qwen2.5-7B-Instruct"
model = AutoModelForCausalLM.from_pretrained(
    model_name, torch_dtype=torch.bfloat16, device_map="auto"
)
tokenizer = AutoTokenizer.from_pretrained(model_name)

# 2. 配置 LoRA
lora_config = LoraConfig(
    r=16,                      # 秩，越大能力越强但越慢
    lora_alpha=32,             # 缩放因子
    target_modules=["q_proj", "v_proj", "k_proj", "o_proj"],
    lora_dropout=0.05,
    task_type="CAUSAL_LM",
)
model = get_peft_model(model, lora_config)
model.print_trainable_parameters()  # 通常只有 0.1-1% 参数可训练

# 3. 准备数据集（对话格式）
dataset = [
    {
        "messages": [
            {"role": "system", "content": "你是游戏中的铁匠NPC..."},
            {"role": "user", "content": "有什么武器卖？"},
            {"role": "assistant", "content": "哈！看看我的好货..."},
        ]
    },
    # ... 几百到几千条这样的对话
]

# 4. 训练
training_args = SFTConfig(
    output_dir="./npc-blacksmith",
    num_train_epochs=3,
    per_device_train_batch_size=4,
    learning_rate=2e-4,
    bf16=True,
    logging_steps=10,
    save_strategy="epoch",
)

trainer = SFTTrainer(
    model=model,
    args=training_args,
    train_dataset=dataset,
    tokenizer=tokenizer,
)
trainer.train()

# 5. 保存 LoRA 权重（通常只有几十MB）
model.save_pretrained("./npc-blacksmith-lora")
```

### RAG 系统全流程

```
用户问题: "黑暗精灵族的起源是什么？"
    ↓
[Query Embedding] → 问题向量化
    ↓
[向量检索] → 从游戏世界观文档中找到相关段落
    ↓
[重排序 Rerank] → 精排最相关的 Top-K 段落
    ↓
[构造 Prompt] → 把检索到的内容 + 问题拼接
    ↓
[LLM 生成] → 基于检索内容回答
    ↓
输出: "根据《世界编年史》，黑暗精灵族起源于..."
```

#### 完整 RAG 实现

```python
from sentence_transformers import SentenceTransformer
import chromadb

# 1. 文档处理
def chunk_document(text, chunk_size=512, overlap=50):
    """把长文档切成重叠的小块"""
    chunks = []
    for i in range(0, len(text), chunk_size - overlap):
        chunks.append(text[i:i + chunk_size])
    return chunks

# 2. 向量化存储
embed_model = SentenceTransformer("BAAI/bge-large-zh-v1.5")
db = chromadb.Client()
collection = db.create_collection("game_lore")

# 加载游戏世界观文档
docs = load_game_documents()  # 你的游戏设定文档
for doc in docs:
    chunks = chunk_document(doc.content)
    embeddings = embed_model.encode(chunks)
    collection.add(
        documents=chunks,
        embeddings=embeddings.tolist(),
        ids=[f"{doc.id}_{i}" for i in range(len(chunks))],
        metadatas=[{"source": doc.title}] * len(chunks),
    )

# 3. 检索 + 生成
def rag_query(question: str, top_k: int = 5):
    # 检索
    query_embedding = embed_model.encode([question])
    results = collection.query(
        query_embeddings=query_embedding.tolist(),
        n_results=top_k,
    )

    # 构造 prompt
    context = "\n---\n".join(results["documents"][0])
    prompt = f"""基于以下游戏世界观资料回答问题。如果资料中没有相关信息，说"这个信息暂不可知"。

资料：
{context}

问题：{question}
回答："""

    # LLM 生成
    response = llm.generate(prompt)
    return response
```

#### 检索优化策略

```
1. Hybrid Search（混合检索）
   BM25（关键词匹配）+ Vector（语义匹配）结合
   适合：专有名词多的游戏场景（"奥利哈刚"BM25 能精确匹配）

2. Reranking（重排序）
   先用向量检索召回 Top-20，再用 Cross-Encoder 精排到 Top-5
   推荐模型：bge-reranker-v2-m3

3. Query Rewriting（查询改写）
   用 LLM 把用户口语化问题改写为更适合检索的形式
   "那个很厉害的剑叫什么" → "游戏中最强传说级武器名称"

4. 分块策略
   - 按语义段落切分（不是固定字数）
   - 保留上下文重叠（overlap）
   - 元数据标注（来源、章节、角色）
```

### Agent 开发

#### ReAct 模式

```
ReAct = Reasoning（推理） + Acting（行动）

循环过程：
1. Thought: 分析当前状况，决定下一步
2. Action: 调用工具执行操作
3. Observation: 观察工具返回结果
4. 重复直到完成任务

示例 - 游戏 GM Agent：
Thought: 玩家投诉说丢了装备，我需要查询日志
Action: query_player_log(player_id="12345", last_hours=24)
Observation: 发现玩家在 2 小时前交易了该装备给另一玩家
Thought: 不是 Bug 导致丢失，是玩家自己交易的
Action: reply_to_player("查询到您2小时前将该装备交易给了玩家xxx，非系统异常")
```

#### Agent 实现框架

```python
class GameAgent:
    def __init__(self, llm, tools):
        self.llm = llm
        self.tools = {t.name: t for t in tools}
        self.memory = []

    def run(self, user_input: str) -> str:
        self.memory.append({"role": "user", "content": user_input})

        for _ in range(10):  # 最多 10 轮推理
            response = self.llm.chat(
                messages=self._build_messages(),
                tools=self._get_tool_schemas(),
            )

            if response.tool_calls:
                # 执行工具
                for call in response.tool_calls:
                    result = self.tools[call.name].execute(call.args)
                    self.memory.append({
                        "role": "tool",
                        "name": call.name,
                        "content": str(result),
                    })
            else:
                # 生成最终回复
                return response.content

        return "思考超时，请简化您的请求。"
```

### 量化技术

```
为什么量化：
FP16 7B 模型 ≈ 14GB 显存
INT4 7B 模型 ≈ 4GB 显存（一张 4060 就能跑）

主流量化方案：
┌──────────┬──────────┬────────────┬──────────────┐
│ 方案     │ 精度损失 │ 速度       │ 适用         │
├──────────┼──────────┼────────────┼──────────────┤
│ GPTQ     │ 小       │ GPU 快     │ GPU 推理     │
│ AWQ      │ 小       │ GPU 更快   │ GPU 推理     │
│ GGUF     │ 可控     │ CPU/GPU    │ llama.cpp    │
│ BitsAndBytes│ 极小  │ 稍慢       │ 微调时省显存 │

```

### 关键优化技术

#### PagedAttention

```
问题：KV-Cache 预分配浪费大量显存（类似内存碎片）
解决：像操作系统虚拟内存一样，按页分配 KV-Cache

效果：
- 显存利用率从 ~50% 提升到 ~95%
- 相同显存支持更多并发请求
- 这是 vLLM 的核心创新

你的直觉：这就是"内存池管理"——你做游戏服务器时可能自己实现过。
```

#### Continuous Batching

```
传统 Batching：等一批请求都完成才处理下一批
Continuous Batching：请求完成一个就填入新的，GPU 永远不空闲

类比：
传统 = 餐厅等所有人吃完才翻台
Continuous = 谁吃完谁走，空位立即坐新客人

效果：吞吐量提升 2-5 倍
```

#### Prefix Caching

```
场景：多个请求共享相同的 System Prompt
优化：System Prompt 的 KV-Cache 只计算一次，所有请求复用

游戏场景完美适用：
所有找铁匠NPC的玩家共享同一个 System Prompt
→ 系统级 KV-Cache 共享，大幅降低首 token 延迟
```
